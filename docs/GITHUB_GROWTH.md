# GitHub Growth Playbook

This is the free traffic plan for OSINTPRO.

## Goal

Make the repository easy to discover, easy to understand and easy to share.

The repository should answer four questions immediately:

1. What is OSINTPRO?
2. Who is it for?
3. How do I try it?
4. Why is it safe and different?

## Repository Search

- Keep the README in English.
- Put high-intent keywords in natural text.
- Put the live demo in the repository homepage field.
- Use a direct repository description:

```text
Passive OSINT workspace for client-ready domain, social and wallet investigation graphs.
```

- Use GitHub topics:
  - `osint`
  - `passive-osint`
  - `domain-intelligence`
  - `web-audit`
  - `web-security`
  - `security-headers`
  - `brand-monitoring`
  - `wallet-osint`
  - `blockchain-analysis`
  - `fraud-investigation`
- `threat-intelligence`
- `cybersecurity`
- `sqlite`
- `python`
- Keep the live demo in the repository homepage field.
- Keep roadmap and issue templates active.

## README Conversion Checklist

- First screen says what the product does.
- Live demo URL is visible near the top.
- Local setup is linked near the top.
- Safety boundary is visible before advanced features.
- Screenshots or sanitized examples are linked.
- Pricing is clear.
- Data sources and costs are documented.
- Roadmap is honest about infrastructure tradeoffs.

## Content To Publish

- Short build-in-public updates after every shipped feature.
- Sanitized screenshots of the Web Audit Lab, Entity Graph and Wallet Trace.
- Example reports for public domains.
- Small educational posts explaining CSP, HSTS, DMARC, wallet tracing and entity graphs.
- Short comparison posts: "raw scanner output vs client-ready evidence graph".
- Short demo clips showing one workflow in under 60 seconds.

## Community Targets

Share only with a passive/intelligence framing:

- indie builder communities
- web agency communities
- blue-team and threat intelligence communities
- OSINT learning groups
- crypto safety and scam-awareness communities
- GitHub project showcases

## Weekly Distribution Routine

Monday:

- Ship or document one visible product improvement.
- Update README, roadmap or docs.
- Open one small issue for the next improvement.

Wednesday:

- Post one build-in-public update on LinkedIn or X.
- Include one screenshot or short workflow.
- Link the live demo and GitHub repo.

Friday:

- Share one educational mini-post in a relevant community.
- Ask for feedback on one specific workflow.
- Avoid generic promotion.

## What Not To Say

- Do not call it an exploit tool.
- Do not promise hacking results.
- Do not imply it can deanonymize wallets.
- Do not advertise brute force, bypasses or offensive payloads.

## Weekly Routine

1. Ship one visible improvement.
2. Update README or docs with the improvement.
3. Post a short build note.
4. Open or close a roadmap issue.
5. Share one sanitized workflow screenshot.
