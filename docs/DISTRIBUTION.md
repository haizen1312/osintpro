# Distribution Plan

OSINTPRO should not depend on GitHub discovery alone. GitHub is useful for trust, documentation and developer visibility, but the first paying users are more likely to come from consultants, agencies, fraud analysts and founders with an immediate reporting problem.

## Primary Positioning

OSINTPRO is a client-ready passive investigation graph.

It helps a consultant or investigator turn public evidence from domains, usernames and wallets into readable findings, relationship maps, exports and recurring monitoring.

## First Audience

Start with one focused audience:

Small security, privacy, GDPR and web agencies that already sell website checks, compliance help, incident response support or client reporting.

They understand the value of:

- recurring domain monitoring
- readable PDF/CSV evidence
- client folders
- plain-English security posture explanations
- lightweight wallet and identity checks when fraud questions appear

## First Channel

Use LinkedIn and direct agency outreach before broad launch platforms.

Weekly motion:

1. Publish one sanitized investigation screenshot or mini case.
2. Explain one practical risk in plain English.
3. Invite agencies to test the live demo.
4. Offer one free passive report for their own domain.
5. Push recurring monitoring and client folders as the paid reason to upgrade.

## Secondary Channels

Use these after the positioning is consistent:

- GitHub: README keywords, topics, sanitized examples and roadmap issues.
- Product Hunt: launch when onboarding, exports and billing feel polished.
- Hacker News: show a technical write-up, not a sales pitch.
- Reddit cybersecurity communities: share educational reports and ask for feedback.
- Crypto safety communities: show wallet tracing as public fraud reconstruction, never deanonymization.

## Messaging

Short pitch:

```text
OSINTPRO turns passive public evidence from domains, usernames and wallets into client-ready investigation graphs, reports and monitoring.
```

What to avoid:

- claiming absolute security
- claiming attribution without verified context
- presenting the product as an exploit scanner
- promising unlimited free infrastructure-heavy usage

## Traffic Goals

Early success should be measured by:

- demo signups
- reports generated per account
- PDF/CSV exports
- client folders created
- monitors attempted by Free users
- Pro checkout clicks
- outreach replies from agencies

Stars are useful, but paid intent is more important.

## Go-To-Market Timeline

| Period | Channel | Concrete motion |
| --- | --- | --- |
| Month 1-2 | Product Hunt prep, Hacker News technical write-up, GitHub docs SEO | Publish one polished passive investigation workflow and collect feedback. |
| Month 3-4 | LinkedIn direct outreach | Contact 100 small agencies with one free passive report offer. |
| Month 5-6 | Reddit, OSINT, cybersecurity and crypto-safety communities | Share educational mini-cases, not generic ads. |
| Ongoing | GitHub issues, roadmap updates, example reports | Keep search-friendly activity visible. |

## Public Traction

Current public traction should stay honest:

- Live demo is deployed.
- Stripe checkout is configured.
- GitHub repository has English docs, topics, issues and discussions enabled.
- First-party conversion signals are tracked in the private admin panel.
- Public signups, paid conversions and MRR should be published only when the numbers are real and stable enough to report.
